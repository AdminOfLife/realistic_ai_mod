RAD to the Win32 Blitting Rescue

Discussion

So, which technology should you use? Bink's built-in software, overlays, off-screen buffers, or 3D blitting?

If your application is 2D, then we include the BinkBuffer API to hide a bunch of the DirectDraw complexity to make switching between the various colorspace options really easy. The Bink player, the ExamWin.C example, and the ExamMSS.C example all use the BinkBuffer API to make displaying Bink frames simple.

Note, however, that the BinkBuffer API is fairly high-level, so if you've already got DirectDraw or running in your application, you may simply want to skip BinkBuffers completely and use the internal Bink software routines to convert directly onto the primary or a secondary surface. See the ExamDDP.C file for an example of this type of playback.

Also note that the BinkBuffer API should not be used when the 3D system is active. While this will work on some 3D cards, older cards (or older drivers) can't blit 2D data over a 3D window, so your video will never be seen.

For 3D applications, we include several different APIs that simplify calling the 3D hardware. In most cases, you don't have to even change the example code. For pixel shader blitting, see the BinkTextures.h API file. For fixed function 3D blitting, see the RAD3D.h API file.

So, using one of these techniques, you should have video up and running in your application quickly and easily!
