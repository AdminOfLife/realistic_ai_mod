Drawing the Bink Video Frames on the Sony PSP

Discussion

Video playback on the Sony PSP is slightly complicated, but not drastically so. The PSP is pretty slow (about half the speed of the PS2), so we can't possibly do the color conversion in software. Unfortunately, the PSP doesn't have register combiners or pixel shaders either, so we needed to figure out another way to use the GPU.

Jason Dorie actually had the good idea to use the palettes combined with multipass blending to do the color conversion. This is great because it's super fast, it looks terrific, and uses minimal memory. Almost like real pixel shaders!

To make things simple, Bink even supplies a little API to do all of the work for you! You just make one call to allocate the textures, and another to draw them. It's really simple and it's almost exactly the same as our pixel shading APIs, so porting will be a snap. And it supports alpha planes directly, so no extra development needs to be done!

The only other complicated thing is that we can't ever be idle during playback on the PSP. That means we need to be decompressing one video frame while the previous one is being rendered. On most platforms, this is the default, but on the PSP, most of the source code examples do exactly the wrong thing - they make you wait for a frame to be rendered before advancing to the next one!

And while this is bad for your game, it's deadly for video playback on a machine that is this slow. Most games will have already written their own page flipping that avoids this problem, but if you haven't, then Bink supplies a simple standalone API to add it. Hey, your game might even get faster! See the "pageflip.h" file for details.

For an complete example of good PSP playback, see the "ExamPSP.CPP" file.
